**Problem:** IpSearchService.php lines 49-51 catch OpenCtiConnectionException, preventing it from reaching the controller's refund logic. Per CONTEXT.md decision: "Refund only on API error (OpenCTI unreachable/timeout)". The exception MUST propagate so the controller can refund.

**Fix in backend/app/Services/IpSearchService.php:**

Change the catch block at lines 49-51 from:
```php
} catch (\App\Exceptions\OpenCtiConnectionException|\App\Exceptions\OpenCtiQueryException) {
    // OpenCTI unavailable or query failed — continue with geo-only results
}
```

To separate the two exception types — re-throw connection exceptions (for refund), but still swallow query exceptions (for graceful degradation):
```php
} catch (\App\Exceptions\OpenCtiConnectionException $e) {
    // Connection failure must propagate to controller for credit refund
    throw $e;
} catch (\App\Exceptions\OpenCtiQueryException) {
    // Query format error — continue with geo-only results
}
```

Rationale: OpenCtiConnectionException means OpenCTI is unreachable (network/timeout) — user should get a refund. OpenCtiQueryException means the query ran but had issues — graceful degradation to geo-only is acceptable (credit charged for valid attempt).

**Verify the unit test:** After this fix, the existing test at line 297 ("search throws exception when OpenCTI is unreachable") should pass as-is because it expects `OpenCtiConnectionException` to propagate — which it now does again.
