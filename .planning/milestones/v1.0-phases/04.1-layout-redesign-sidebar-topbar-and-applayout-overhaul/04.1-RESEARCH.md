# Phase 04.1: Layout Redesign -- Sidebar, Topbar, and AppLayout Overhaul - Research

**Researched:** 2026-03-13
**Domain:** React UI layout redesign (sidebar navigation, topbar, responsive shell)
**Confidence:** HIGH

## Summary

This phase is a frontend-only UI overhaul of the application shell (Sidebar, Topbar, AppLayout). The existing codebase uses React 19, React Router DOM 7, Tailwind CSS 3, and Lucide React icons. No new libraries are needed -- all changes use the existing stack. The core work involves: (1) replacing the grouped NAV_ITEMS with a flat list, (2) adding collapsible sidebar with localStorage persistence, (3) making the sidebar responsive with a mobile drawer, (4) adding auth-aware nav states (locked items for guests), (5) simplifying the topbar (remove search trigger, add avatar dropdown with Manage Profile/Logout), and (6) restructuring routes to make IP Search publicly accessible.

The existing code is well-structured with clear separation: `Sidebar.jsx`, `Topbar.jsx`, `AppLayout.jsx` as the layout shell, `ProtectedRoute.jsx` for auth gating, and `mock-data.js` for the NAV_ITEMS data. The `useAuth()` hook already provides `isAuthenticated` which is the key signal for conditional rendering. The main architectural change is moving `/ip-search` (renamed from `/ioc-search`) outside of `ProtectedRoute` while still wrapping it in `AppLayout`.

**Primary recommendation:** Restructure routes in App.jsx so AppLayout wraps both protected and public-with-layout routes, with IP Search accessible to guests. Manage sidebar collapse state in AppLayout via useState + localStorage, passing it down to Sidebar and Topbar via props or context.

### Pattern 1: Sidebar Collapse State Management

**What:** Centralize sidebar collapse state in a custom hook with localStorage persistence.
**When to use:** AppLayout initializes the hook, passes state down to Sidebar and Topbar.

```jsx
// hooks/useSidebarCollapse.js
import { useState, useCallback } from 'react';

const STORAGE_KEY = 'sidebar-collapsed';

export function useSidebarCollapse() {
  const [collapsed, setCollapsed] = useState(() => {
    try {
      return localStorage.getItem(STORAGE_KEY) === 'true';
    } catch {
      return false;
    }
  });

  const toggle = useCallback(() => {
    setCollapsed(prev => {
      const next = !prev;
      try { localStorage.setItem(STORAGE_KEY, String(next)); } catch {}
      return next;
    });
  }, []);

  return { collapsed, toggle };
}
```

### Pattern 3: Route Restructuring for Public IP Search

**What:** Move IP Search out of ProtectedRoute while keeping it inside AppLayout.
**When to use:** App.jsx route configuration.

```jsx
// App.jsx -- restructured routes
<Routes>
  <Route index element={<LandingPage />} />

  {/* Guest-only routes (no AppLayout) */}
  <Route element={<GuestRoute />}>
    <Route path="/login" element={<LoginPage />} />
    <Route path="/register" element={<RegisterPage />} />
    {/* ... */}
  </Route>

  {/* Public pages with AppLayout (accessible to all) */}
  <Route element={<AppLayout />}>
    <Route path="/ip-search" element={<IpSearchPage />} />
  </Route>

  {/* Protected pages with AppLayout (auth required) */}
  <Route element={<ProtectedRoute />}>
    <Route element={<AppLayout />}>
      <Route path="/dashboard" element={<DashboardPage />} />
      <Route path="/threat-map" element={<ThreatMapPage />} />
      {/* ... */}
    </Route>
  </Route>
</Routes>
```

**Important:** This creates two `<AppLayout />` instances. To avoid this, a better pattern is to wrap AppLayout around both and handle auth checks per-route:

```jsx
{/* All app routes with layout */}
<Route element={<AppLayout />}>
  {/* Public route */}
  <Route path="/ip-search" element={<IpSearchPage />} />

  {/* Protected routes */}
  <Route element={<ProtectedRoute />}>
    <Route path="/dashboard" element={<DashboardPage />} />
    <Route path="/threat-map" element={<ThreatMapPage />} />
    {/* ... */}
  </Route>
</Route>
```

This is the preferred approach -- single AppLayout instance, ProtectedRoute nested inside for auth-gated routes.

### Pattern 4: Hover-Expand on Collapsed Sidebar

**What:** When sidebar is collapsed (64px rail), hovering temporarily expands it as an overlay.
**When to use:** CSS + React state for hover detection.

```jsx
// Sidebar hover-expand pattern
const [hovered, setHovered] = useState(false);

// The sidebar element:
<aside
  onMouseEnter={() => collapsed && setHovered(true)}
  onMouseLeave={() => setHovered(false)}
  className={`
    fixed left-0 top-0 h-screen z-40 flex flex-col
    transition-all duration-300 ease-in-out
    ${collapsed && !hovered ? 'w-16' : 'w-[260px]'}

  style={{
    // Glassmorphism
    background: 'rgba(15, 17, 23, 0.8)',
    backdropFilter: 'blur(20px)',
    borderRight: '1px solid rgba(30, 32, 48, 0.8)',
  }}
>
```

### Pattern 5: Mobile Drawer with Overlay

**What:** On small screens, sidebar hidden by default, opened via hamburger as overlay with backdrop.
**When to use:** Breakpoint-based (< 768px recommended, or lg: 1024px for richer layouts).

```jsx
// AppLayout manages mobile drawer state
const [mobileOpen, setMobileOpen] = useState(false);

// Topbar shows hamburger on mobile
<button className="lg:hidden p-2" onClick={() => setMobileOpen(true)}>
  <Menu className="w-5 h-5" />
</button>

// Sidebar receives mobileOpen prop
// On mobile: sidebar is position-fixed overlay with backdrop
// On desktop: sidebar is normal fixed sidebar
```

### Anti-Patterns to Avoid
- **Duplicating AppLayout in routes:** Do NOT create separate AppLayout wrappers for public vs protected routes. Use single AppLayout with nested ProtectedRoute.
- **Mutating localStorage on every render:** Use callback initializer in useState to read localStorage once, write only on toggle.
- **Inline glassmorphism styles everywhere:** Use existing `.glass-panel` or create a single `.glass-sidebar` class in CSS. The project already has glassmorphism utility classes.
- **Hard-coding sidebar width in multiple places:** The Topbar `left-[260px]` and main-content `ml-[260px]` must be dynamic based on collapse state. Use CSS variables or pass width as prop/context.

## Common Pitfalls

### Pitfall 1: Topbar Left Offset Not Matching Sidebar Width
**What goes wrong:** When sidebar collapses to 64px, the Topbar still has `left-[260px]` hard-coded, leaving a gap.
**Why it happens:** Current Topbar has `fixed top-0 left-[260px]` and main-content has `ml-[260px]` in CSS.
**How to avoid:** Pass `collapsed` state to Topbar and main-content. Use dynamic class: `left-[260px]` vs `left-16` (64px). Same for `ml-[260px]` vs `ml-16`. On mobile, both should be `left-0` / `ml-0`.
**Warning signs:** Visual gap between sidebar and topbar, or content hidden behind sidebar.

### Pitfall 2: Hover-Expand Conflicts with Click Events
**What goes wrong:** Hover-expand triggers while user is clicking a nav item, causing jittery UX.
**Why it happens:** mouseEnter/mouseLeave events fire rapidly during interaction.
**How to avoid:** Add a small delay (150ms) before expanding on hover. Use `setTimeout` with cleanup on mouseLeave. Only enable hover-expand when `collapsed` is true.
**Warning signs:** Sidebar flickers between collapsed and expanded on mouse movement.

### Pitfall 3: Mobile Drawer Z-Index Conflicts
**What goes wrong:** Mobile sidebar drawer appears behind the NotificationDrawer or other overlays.
**Why it happens:** Current z-index hierarchy: sidebar z-40, topbar z-30, notification drawer z-60.
**How to avoid:** Mobile sidebar overlay should use z-50 (between topbar and notification drawer). Backdrop overlay behind it at z-49 or same z-50.
**Warning signs:** Clicking through the sidebar overlay, or notification drawer covering sidebar.

### Pitfall 4: NavLink Active State on Renamed Routes
**What goes wrong:** After renaming /ioc-search to /ip-search, old bookmarks or landing page CTAs break.
**Why it happens:** Route path changed but references not updated everywhere.
**How to avoid:** Search entire codebase for `/ioc-search` references. Key locations: LandingPage.jsx CTAs, mock-data.js NAV_ITEMS, Topbar.jsx PAGE_NAMES, App.jsx route definitions.
**Warning signs:** 404 on navigation, wrong nav item highlighted.

### Pitfall 5: ProtectedRoute Redirect Loop for Guests on IP Search
**What goes wrong:** Guest visits /ip-search, ProtectedRoute redirects to /login, but /ip-search should be public.
**Why it happens:** IP Search is still nested under ProtectedRoute in App.jsx.
**How to avoid:** Route /ip-search MUST be outside ProtectedRoute wrapper. Use the single AppLayout pattern with nested ProtectedRoute only around auth-required routes.
**Warning signs:** Guests redirected to login when clicking "IP Search" from landing page.

### Pitfall 6: Settings Submenu State Persistence
**What goes wrong:** Settings submenu expand/collapse state resets on navigation.
**Why it happens:** Sidebar re-renders on route change, losing local state.
**How to avoid:** Sidebar component is mounted once inside AppLayout and persists across route changes (Outlet pattern). The useState for submenu open/close should survive. Verify Sidebar is NOT re-mounted on navigation.
**Warning signs:** Settings submenu closes when navigating to a different page.
