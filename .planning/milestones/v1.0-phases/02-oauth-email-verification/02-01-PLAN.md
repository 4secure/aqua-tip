  <action>
    Follow the invokable single-action controller pattern established in Phase 1.

    1. **Modify RegisterController.php**: Add `use Illuminate\Auth\Events\Registered;` and dispatch `event(new Registered($user));` immediately after `User::create()`, BEFORE `Auth::login($user)`. This triggers the automatic verification email via Laravel's `SendEmailVerificationNotification` listener.

    2. **Create ForgotPasswordController.php** (invokable):
       - Accept `ForgotPasswordRequest` (validates `email` as required|email)
       - FIRST check if the user has `oauth_provider` set. If so, return 422 with message: "This account uses {provider} to sign in. Please use {provider} instead."
       - Otherwise call `Password::sendResetLink($request->only('email'))`
       - Return 200 with `__($status)` on success, 422 with `__($status)` on failure
       - Do NOT use `stateless()` or anything custom -- the Password broker handles throttling (60s per email, configured in auth.php)

    3. **Create ResetPasswordController.php** (invokable):
       - Accept `ResetPasswordRequest` (validates token, email, password with confirmation, password uses same rules as RegisterRequest: `Password::min(8)->mixedCase()->numbers()`)
       - Use `Password::reset()` with callback that `forceFill`s password (Hash::make) and sets a new remember token (Str::random(60)), then fires `event(new PasswordReset($user))`
       - Return 200 on success, 422 on failure (expired token, invalid email, etc.)
       - Per user decision: do NOT auto-login after reset. Just return success message.

    4. **Create VerifyEmailController.php** (invokable):
       - Use `EmailVerificationRequest` (auto-validates signed URL + auth)
       - If already verified, return 200 with "Email already verified."
       - Otherwise call `$request->fulfill()` and return 200 with "Email verified successfully."

    5. **Create ResendVerificationController.php** (invokable):
       - Require `auth:sanctum` middleware (applied at route level)
       - If user already verified, return 200 with "Email already verified."
       - Otherwise call `$request->user()->sendEmailVerificationNotification()` and return 200 with "Verification link sent."

    6. **Create ForgotPasswordRequest.php**: Validates `email` as `required|email`.

    7. **Create ResetPasswordRequest.php**: Validates `token` (required|string), `email` (required|email), `password` (required|confirmed) with same `Password::min(8)->mixedCase()->numbers()` rule from RegisterRequest.

    8. **Modify AppServiceProvider.php boot()**: Add URL customization AFTER the existing RateLimiter code:
       - `VerifyEmail::createUrlUsing()` -- generate URL pointing to frontend: `config('services.frontend_url') . '/verify-email?verify_url=' . urlencode($url)`
       - `ResetPassword::createUrlUsing()` -- generate URL pointing to frontend: `config('services.frontend_url') . '/reset-password?token=' . $token . '&email=' . urlencode($user->email)`
       - Import `Illuminate\Auth\Notifications\VerifyEmail` and `Illuminate\Auth\Notifications\ResetPassword` and `App\Models\User`
  </action>

  <action>
    Update routes/api.php to add new endpoints and enforce email verification on protected routes.

    1. **Add imports** at top: ForgotPasswordController, ResetPasswordController, VerifyEmailController, ResendVerificationController.

    2. **Add guest-only password reset routes** inside the existing `throttle:auth` middleware group (or a new group if cleaner):
       ```
       Route::post('/forgot-password', ForgotPasswordController::class)->name('password.email');
       Route::post('/reset-password', ResetPasswordController::class)->name('password.update');
       ```
       These are guest routes -- no auth required. The throttle:auth rate limiter is fine for these too.

    3. **Add email verification routes** inside the `auth:sanctum` group:
       ```
       Route::get('/email/verify/{id}/{hash}', VerifyEmailController::class)
           ->middleware('signed')
           ->name('verification.verify');
       Route::post('/email/verification-notification', ResendVerificationController::class)
           ->middleware('throttle:6,1')
           ->name('verification.send');
       ```

    4. **Add `verified` middleware to protected feature routes**: Modify the existing `auth:sanctum` group so that `/user` and `/logout` also require `verified`. Use `middleware(['auth:sanctum', 'verified'])` for the group.

       IMPORTANT: The register and login routes must NOT require verified. The verification routes themselves (verify + resend) need `auth:sanctum` but NOT `verified` (otherwise unverified users cannot verify). Structure the routes as:

       ```php
       // Guest auth routes
       Route::middleware('throttle:auth')->group(function () {
           Route::post('/register', RegisterController::class);
           Route::post('/login', LoginController::class);
           Route::post('/forgot-password', ForgotPasswordController::class)->name('password.email');
           Route::post('/reset-password', ResetPasswordController::class)->name('password.update');
       });

       // OAuth redirect (no auth needed)
       Route::get('/auth/{provider}/redirect', [SocialAuthController::class, 'redirect']);

       // Auth required but NOT verified (for unverified users to verify)
       Route::middleware('auth:sanctum')->group(function () {
           Route::get('/email/verify/{id}/{hash}', VerifyEmailController::class)
               ->middleware('signed')
               ->name('verification.verify');
           Route::post('/email/verification-notification', ResendVerificationController::class)
               ->middleware('throttle:6,1')
               ->name('verification.send');
       });

       // Auth + verified required (protected feature routes)
       Route::middleware(['auth:sanctum', 'verified'])->group(function () {
           Route::get('/user', UserController::class);
           Route::post('/logout', LogoutController::class);
       });
       ```

       NOTE: The `verified` middleware alias should already exist in Laravel 12 via `EnsureEmailIsVerified`. If it does not, register it in `bootstrap/app.php` using `->withMiddleware(function (Middleware $middleware) { $middleware->alias(['verified' => \Illuminate\Auth\Middleware\EnsureEmailIsVerified::class]); })`. Check if `\Illuminate\Auth\Middleware\EnsureEmailIsVerified` is already aliased as `verified` by running `php artisan route:list` after adding the route. If the middleware alias is missing, add it.

    5. **Verify existing tests still pass**: The existing Registration/Login/Logout tests should still work. Registration creates a user and logs them in -- the response is 201 from the register endpoint itself (not a protected route). Login returns 200. The `/api/user` call in tests will now require the user to be verified. Check existing tests -- if they create users without `email_verified_at`, they may need updating. If so, add `'email_verified_at' => now()` to test user factories or directly in test setup.
