Backend adjustments required before frontend auth integration can function.

All must be fixed before any frontend work.

Output: Updated backend routes, controllers, and resources. New migration for phone/onboarding columns. New endpoints for code verification and onboarding. Updated OAuth callback redirect logic.

    3. **RegisterRequest.php**: Change `name` rule from `['required', 'string', 'min:2', 'max:255']` to `['nullable', 'string', 'min:2', 'max:255']`. Remove `'confirmed'` from password rule (keep `Password::min(8)->mixedCase()->numbers()`).

    4. **RegisterController.php**: When `$request->name` is null, set name to the local part of the email (e.g., `explode('@', $request->email)[0]`). The rest stays the same -- the `Registered` event still fires which triggers the verification email.

    5. **User.php**: Add `phone` and `onboarding_completed_at` to `$fillable`. Add `'onboarding_completed_at' => 'datetime'` to casts.

  <action>
    1. **VerifyEmailWithCode notification**: Create a custom notification that replaces the default VerifyEmail. It should generate a 6-digit random numeric code, store it in cache with key `email_verify_code:{user_id}` and 15-min TTL, AND include the standard signed verification link. The email should show both the code prominently and the clickable link. Register this as the verification notification in AppServiceProvider or by overriding `sendEmailVerificationNotification()` on the User model.

    2. **VerifyEmailCodeController.php**: Invokable controller. Route: `POST /api/email/verify-code`. Requires `auth:sanctum`. Accepts `{ code: "123456" }`. Validates code is `required|string|size:6`. Retrieves cached code via `Cache::get("email_verify_code:{$user->id}")`. If match: mark user verified (`$user->markEmailAsVerified()`), delete cache key, return 200 `{ message: "Email verified successfully." }`. If no match or expired: return 422 `{ message: "Invalid or expired verification code." }`.

    3. **OnboardingController.php**: Invokable controller. Route: `POST /api/onboarding`. Requires `auth:sanctum`. Accepts `{ name: "John", phone: "+1234567890" }`. Validates name (required, string, 2-255) and phone (required, string, 5-20). Updates user name, phone, and sets `onboarding_completed_at` to `now()`. Returns updated UserResource with 200.

    4. **routes/api.php**: Add to the `auth:sanctum` group:
       - `Route::post('/email/verify-code', VerifyEmailCodeController::class);`
       - `Route::post('/onboarding', OnboardingController::class);`

    5. **SocialAuthController.php callback()**: After `Auth::login($user)`, check if user has completed onboarding: `if (!$user->phone || !$user->onboarding_completed_at)`. If not onboarded, redirect to `{frontendUrl}/get-started`. If onboarded, redirect to `{frontendUrl}/dashboard` (existing behavior).

    6. **Tests**: Write Pest feature tests:
       - VerifyEmailCodeTest: valid code verifies user, expired/wrong code returns 422, already-verified user gets appropriate response
       - OnboardingTest: valid name+phone sets onboarding_completed_at, missing fields return 422, updates user name
  </action>

    - Verification email includes both 6-digit code and clickable link
    - All new tests pass
  </done>
